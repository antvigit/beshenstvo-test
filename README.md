# Beshenstvo Test — автотесты для калькулятора прививок

Автотесты для сайта [beshenstvo.pro](https://beshenstvo.pro/) — проверка корректности отображения дат вакцинации.

Проект включает:
- Автотесты на Java + Selenium + JUnit 5.
- Allure-отчёты.
- Telegram-бота для удалённого запуска тестов и получения отчётов.

🧱 Архитектура проекта: ООП, POM, SOLID

Проект построен на фундаментальных принципах, которые делают код надёжным и готовым к развитию.

1. Объектно-ориентированное программирование (ООП):

   Наследование: Класс VaccinationPage расширяет абстрактный класс BasePage, который содержит общую логику для всех страниц (драйвер, ожидания). Это позволяет не дублировать код.

   Инкапсуляция: Все элементы страницы (WebElement) объявлены как private. Доступ к ним осуществляется только через публичные методы, что защищает данные от случайного изменения.

   Полиморфизм: Классы используют интерфейс IPage и абстракцию WebDriver. Это позволяет легко подменять одну реализацию на другую (например, ChromeDriver на FirefoxDriver) без изменения кода тестов.

2. Page Object Model (POM) + Page Factory:

   Каждая страница приложения — это отдельный класс (VaccinationPage), который хранит в себе все локаторы и методы взаимодействия с элементами этой страницы.

   Для инициализации элементов используется паттерн Page Factory с аннотацией @FindBy. Это упрощает и ускоряет работу с веб-элементами.

3. Принципы SOLID в проекте:
   Принцип	Что означает	Как реализован в проекте
   S (Single Responsibility)	У класса должна быть только одна причина для изменения.	VaccinationPage отвечает только за логику страницы, PageLocators — только за локаторы, VaccinationCalculatorTest — только за сценарии тестов.
   O (Open/Closed)	Классы должны быть открыты для расширения, но закрыты для модификации.	Можно добавить новую страницу (ResultPage), создав класс-наследник BasePage, не изменяя существующие классы.
   L (Liskov Substitution)	Объекты в программе должны быть заменяемы на экземпляры их подтипов.	Мы используем интерфейс WebDriver. В тестах можно заменить ChromeDriver на FirefoxDriver или любой другой драйвер, и тесты продолжат работать, потому что они зависят от абстракции, а не от конкретной реализации.
   I (Interface Segregation)	Не следует создавать «толстые» интерфейсы.	Интерфейс IPage содержит только два метода: open() и waitForPageLoaded(). Он не перегружен лишними методами, которые могут не понадобиться другим страницам.
   D (Dependency Inversion)	Зависимости должны строиться на абстракциях, а не на конкретиках.	Все страницы получают WebDriver через конструктор (BasePage). Они не создают его сами, а получают готовую абстракцию, что делает код гибким и тестируемым.

🧪 Тесты

В проекте есть один стабильно работающий тест. Остальные тесты временно отключены или падают, так как на целевом сайте отсутствуют формы для их проверки.
Тест	Что проверяет	Статус
shouldValidateVaccinationDates	Наличие дат, их формат ДД.ММ.ГГГГ, хронологический порядок и интервалы (3, 4, 7, 16, 60 дней)	✅ Проходит
testValidNameCyrillic	Ввод имени на кириллице	❌ Падает
testEmptyName	Ошибка при пустом поле	❌ Падает
testNameWithSpecialChars	Ошибка при спецсимволах	❌ Падает
testVeryLongName	Ошибка при превышении лимита	❌ Падает
testInvalidDate	Ошибка при невалидной дате	❌ Падает

## 🤖 Telegram-бот

Бот позволяет запускать тесты удалённо и получать результат в Telegram.

**Команды:**

- `/start` — приветствие.
- `/run` — запустить тесты.

**Что приходит:**

1. Статус запуска (🔄)
2. Ожидание завершения (⏳)
3. Скриншот Allure-отчёта (📊)
4. Ссылка на полный отчёт в GitHub Actions (🔗)
5. Финальное сообщение с результатом (✅)
